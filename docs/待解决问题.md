## 一、代码补全回写 IDE 问题（待验证）

**状态**：待用户实测确认（2026-06-07 记）

### 现象
开代理后，IDE 行内代码补全（ghost text）可能不回写。

### 已排查/排除
- 代理转发请求体 100% 完整（字节数、MD5 一致，无截断）
- 响应流式实时透传正常（非缓冲后一次性吐出）
- 已对 `*/completions*` 启用「低延迟直通」：不强制 gzip、关 Nagle、强制 `Connection: close`
- 抓到的一条补全报文模型本身返回空（`text:""`, `finish_reason:"stop"`），可能是该位置无补全，**不能证明是代理的问题**

### 决定性验证步骤
1. IDE 里敲**确定会出补全**的代码（不开代理能看到灰色 ghost text）
2. **开代理**触发同样补全
3. 点该条 → SSE 格式解析 → 看「拼接正文」：
   - **正文有内容但 IDE 没插入** → IDE 端问题（延迟/被取消/检测到代理），工具侧难解；但统计采集不受影响
   - **正文为空** → 上游收到正确 prompt 却返回空，需对比完整请求头定位

### 接口信息备忘
- 补全接口：`POST http://codebuddy.pab.com.cn/v2/completions`（HTTP 明文）
- 格式：`text_completion`，`choices[].text`（非 chat 的 delta.content）
- 请求头：`connection: close` / `proxy-connection: close`、`x-agent-intent: CodeCompletion`
- 请求体：`stream:true`, `stop:["\n\n","\n\n\n"]`, model `codewise-7b`

## 三、已知边界（被动抓包固有限制，无法修复）

排查"上送内容比实际写入的代码少"时先对照这里：

1. **`replace_in_file` 只有改动块**：报文里只有 `old_str`/`new_str`，合并后的**整文件是 IDE
   在本地磁盘拼出来的，不经过网络**，抓包物理上取不到。上送 `new_str` 恰是"本轮生成的代码"
   ——对代码生成量统计这是**正确口径**（整文件反而会把旧代码算进生成量）。若业务要"采纳后
   的最终完整文件"，只能走 IDE 端插件埋点。
2. **`execute_command` 写文件不计**：模型通过 shell 命令写入的内容在本地执行，报文里只有命令
   字符串。result 的工具留痕会记录该命令（`> 调用 execute_command: {...}`），但不算代码。
3. **验证手段**：管理员开「调试落盘(JSONL)」，`responseRaw` 存了 SSE 原文，可用
   `TRANSFORMS.sseCodeAll(responseRaw)` 重放比对上送值是否一致（2026-07 实测 11/11 逐字节一致）。
