> 依据 v1.0 文档（2026-05-20）。接口：`POST /api/userBehavior/saveRecord`，鉴权头 `apiToken`。
> 抓包样例：CodeBuddy IDE → `POST /v2/chat/completions`，OpenAI 请求体 + SSE 流式响应（model=dsv4）。
> 报文头一律小写：`x-user-id`、`x-conversation-id`、`x-conversation-message-id`、`x-ide-name`、`x-ide-version` 等。

### ⚠️ 坑：Integer 字段必须带数值 transform（否则整包被接口拒）

`codeLines` / `codeSize` / `promptSize` 等 **Integer 字段**若映射成裸 `resText`/`reqText`
（`{source:"resText"}` 而**漏了 transform**），取到的就是**整段 SSE 响应/请求原文（字符串）**，
发给接口的 Integer 字段 → 接口 Jackson 报：

```
Cannot deserialize value of type java.lang.Integer from String "data:..."
（HttpMessageNotReadableException / 中转透传为 Message-Unreadable）
```

**整包被拒**，且这是解析阶段失败，`behavior_raw_inbox` 也未必留痕。

- **真实根因（2026-07-01 定位并修复）**：不是手动加，而是**前端保存时把 transform 弄丢了**——
  transform 下拉选项列表 `MAP_TRANSFORMS` 当时**漏了 `sseCodeLines`/`sseCodeSize`**。
  于是「重置为预设」把 codeLines 渲染进表格时，下拉框没有 `sseCodeLines` 选项 → 显示为空 →
  「保存」时 `collectMapping()` 读到空 transform → codeLines 被存成裸 `{source:"resText"}` →
  上送发原始响应文本字符串 → 接口 `Cannot deserialize Integer from String`。**反复重置也修不好，因为每次保存都丢。**
- **已修复**：① `MAP_TRANSFORMS` 补全 `sseCodeLines`/`sseCodeSize`；② 下拉渲染**兜底包含当前值**
  （即使某 transform 不在列表也不丢）；③ const 纯整数按数字存（`isStatistics` "1"→1，不再发字符串）。
- **快速定位**：任何上送异常，看项目根 `last-upload-debug.json` 的 `fields` 段——
  每个字段的**实际类型 + 值预览**一目了然，Integer 字段若显示成 `string` 就是这个坑。

## ★ 字段映射本地验证：behavior-validator（Java 项目）

**`D:\code\behavior-validator`** 是一个极简 Spring Boot 项目，**代码来源于内网埋点服务
`D:\code\user-behavior-track`**（saveRecord 的 DTO / Jackson 强类型 / JSR-303 / 接口感知校验 / 异常处理
全部提炼自它，字段类型与校验规则**完全一致**），只剥离 DB/MQ/Apollo/转发等依赖，**专门用来验证
proxy-inspector 的字段映射与取值类型**，不依赖内网/数据库。
> `user-behavior-track` 若改了字段或校验规则，需**同步回** behavior-validator，二者保持一致才有验证意义。

**用法（端到端，推荐）**：启动它（`http://127.0.0.1:18080`）→ proxy-inspector「上送设置」把
**上送地址设为 `http://127.0.0.1:18080/api`**（behavior 模式，末尾带 /api）→ 正常抓包上送，直接在 proxy-inspector
的「上送队列」看结果：

- 绿色「已送达」= 字段/类型全合规（校验器回 `code:"0"`）；
- 红色失败 = `lastError` 直接显示哪个字段/类型不对（校验器回 `code:"1"` + msg，含 Jackson 根因，
  如 `Cannot deserialize Integer from String ... codeLines`）——**这句正是内网线上被隐藏、平时看不到的**。

> **约定：以后 proxy-inspector 改任何与上送/字段映射相关的东西，都先指向 behavior-validator 回归验证**，
> 确认字段类型/必填/接口感知校验都过，再切回真实埋点地址。校验器与官方接口同契约（HTTP 200 + `code`）。
> 详见 `behavior-validator/README.md`。

## 六、⚠️ 改了代码默认映射却"不生效"？（重要陷阱）

`src/reporter-mapping.js` 的 `DEFAULT_MAPPING` 只是**首次启动**时写入 `settings.json` 的默认值。
一旦 `settings.json` 生成，运行时用的是**它里面持久化的 `reporterTargets`**，
**不再读代码里的默认值**。所以你改了代码的字段映射（如 `requestId` 的取值），
若 `settings.json` 已存在，界面/上送仍用旧的，表现为"没改动"。

让新映射生效，三选一：

1. 面板「字段映射」tab → **「重置为预设」**（读 `mapping-presets/codebuddy-sse.json`）→ 保存；
2. 手动改 `settings.json` 里的 `reporterTargets`；
3. 停服务删除 `settings.json` 的 `reporterTargets` 字段（或整个文件）→ 重启重新生成。

> 同理：分发包不带 `settings.json`，新机器首启会用最新代码默认值；但**老机器**升级时要注意上面这点。
