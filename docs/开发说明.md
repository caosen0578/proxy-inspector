# 资金同业代码解析工具 - 项目说明文档

> 最后更新：2026-06-24

---

## 一、项目概述

- **工具名称**：资金同业代码解析工具（可在「上送设置」里修改）
- **工作目录**：`D:\code\proxy-inspector`
- **本质**：Node.js HTTP/HTTPS MITM 代理 + Web 管理面板
- **用途**：拦截 CodeBuddy IDE 插件与 AI 服务之间的流量，提取 AI 产出的代码，自动上送至行内用户行为埋点接口，统计个人代码生成率
- **面板地址**：http://127.0.0.1:28900
- **启动方式**：`start.cmd`（Windows）；打包：`pack.cmd`

## 三、核心模块

### `src/reporter-mapping.js` — 报文字段映射 & 代码提取

**两个 CodeBuddy 接口的提取规则：**

#### 接口一：`/v2/completions`（行内代码补全 CodeCompletion）

SSE 格式：每个 chunk 用 `choices[0].text`（裸代码，无 ``` 围栏）

| 埋点字段 | 提取规则 |
|---------|---------|
| result | `sseFullReply`（补全接口无工具调用，行为 = 旧 `sseContent`：拼接所有 `choices[0].text`） |
| acceptResult | `sseCodeAll`（extractGeneratedCode）；补全接口走裸代码路径 = 全量 `choices[0].text` |
| codeLines | 提取代码（同 acceptResult）的行数 |
| codeSize | 提取代码（同 acceptResult）的 UTF-8 字节数 |
| modelName | 任意 chunk 的 `model` 字段 |
| promptTokens | 最后含 `usage` 的 chunk 的 `usage.prompt_tokens` |
| completionTokens | 最后含 `usage` 的 chunk 的 `usage.completion_tokens` |
| finishReason | 最后含 `finish_reason` 的 chunk 的 `choices[0].finish_reason` |
| language | 请求体 `extra.language` |
| filePath | 请求体 `extra.file_name` |
| repository | 请求体 `extra.repo_name` |
| prompt | 请求体 `prompt` 字段（代码上文）——`promptAny` 自动识别 |
| promptSize / promptMd5 | 对 prompt 算 UTF-8 字节数 / MD5 |

#### 接口二：`/v2/chat/completions`（Craft/Agent 对话模式）

SSE 格式：`choices[0].delta.content`（说明文字）+ 可能含 `delta.tool_calls`（工具调用）

`extractGeneratedCode` 取值（2026-07 起为**并集**，早期"三选一短路"会丢同轮旁白里的代码）：

1. **写入型工具调用**：取 `new_str ?? content ?? code ?? newText`。
   - `write_to_file.content` = 完整文件代码
   - `replace_in_file.new_str` = 替换后代码片段
   - 只读工具（`read_file` / `read_lints` / `search`）无这些字段 → 自动跳过
   - **抗模型精度问题**（`toolCallsAll` + `splitJsonObjects`）：**代码提取完全不依赖 `id`/`name`/`index`**——把所有 `arguments` 分片**全量拼接** → **括号配平**切出完整 JSON 对象（带起始偏移）→ 提取代码 → **按文本去重**。这样名字黏连（`write_to_filewrite_to_file`）、index 恒 0、content 被拆段、整次重复发送都能正确还原。`name` 仅用于 result 留痕展示：按偏移回配"最近一个 name"，黏连名折叠，配不上也不影响代码提取。详见 [CodeBuddy代码提取规则.md](CodeBuddy代码提取规则.md)。
   - 多个不同的写入工具调用用 `\n\n` 拼接
2. **delta.content ``` 围栏代码块**：**并入**工具代码（按代码文本去重）；无工具、无围栏且为补全流时取裸代码
3. **全空 → 返回空**：`''` → `reporter.push()` 跳过不入队

`result`（`sseFullReply`）：旁白 + 本轮**所有工具调用按流式顺序留痕**——写码工具为
`[工具名] 文件 \`路径\`：`+``` 围栏块（不截断、按 code 去重），其余工具为 `> 调用 工具名: {参数摘要}`
单行（单值>120字截断标注总长、整行>300字截断；全文可查调试 JSONL）。时序可排查
"先读了什么→改了什么→跑了什么命令"。纯聊天/补全无工具时行为 = 旧 `sseContent`。

**两接口通用字段：**

| 字段 | 来源 |
|------|------|
| sessionId | 请求头 `x-conversation-id` |
| requestId | 请求头 `x-conversation-message-id` |
| scope | 请求头 `x-ide-type` |
| pluginVersion | 请求头 `x-ide-version`（由 `config.reporterTriggerVersion` 决定） |
| createdBy | 系统用户名（UM号），`os.userInfo().username`，**统一转小写**（显示与上送一致） |
| type | 固定 `"CODE_CHAT"` |
| cost | `record.duration`（毫秒） |
| apiUrl | `record.url` |
| prompt | `promptAny`：对话取 `messages` 拼接 / 补全取 `prompt` 字段（两接口兼容） |
| promptSize / promptMd5 | 对 prompt 算 UTF-8 字节数 / MD5（纯计算，两接口适用） |

### `src/reporter.js` — 上送队列

- **内容过滤**（`_passesContentFilter`，入队 + 发送前都校验）：`result` 或 `acceptResult` **任一为空 → 不上送**。
  - 入队时不过 → `push()` 不入队；
  - 发送前不过 → `_flush()` **直接丢弃**该条（不发、不重试、不计历史），打日志。
  - 发送前再校验的意义：挡住"旧规则入队 / `reporter-queue.json` 持久化"的历史脏条目。
- **业务成功判定**：HTTP 200 ≠ 成功；需检查响应体 `data.code === '0'`（或 `success === true`）；`code:"1"` 视为失败 → 自动重试（最多 `REPORTER_MAX_ATTEMPTS` 次）
- **上送超时**：单次发送 `axios` 超时取 `cfg.reporterTimeoutMs`（配置项，默认 `config.REPORTER_TIMEOUT_MS`=15000，面板「上送设置」可改，仅管理员，范围 1000~120000）。**须 > 后端最坏响应时间**，否则后端慢成功时客户端先超时→误判失败→重发→重复上送（后端按 `requestId` 幂等兜底，调大此值从源头减少重发）
- **单条重试**：`retryOne(qid)` 复位指定条并立即发送（UI 每行「重试」按钮）
- **持久化**：队列落盘 `reporter-queue.json`，应用重启后继续发送
- **已送达历史**：内存保留最近 100 条（含完整 entry），可同等预览已送达报文
- **上送调试落盘**（`dumpUploadDebug`）：behavior 模式每次发送前，把**实际上送的原始字节 + 逐字段类型 + JSON 合法性**写入项目根 `last-upload-debug.json`（覆盖写，仅最近一条）。排查"接口反序列化失败/传值类型不对"时直接看 `fields` 段：Integer 字段若显示成 `string` 即命中"整数字段漏 transform"坑（见下）。

> ⚠️ **field 源两遍解析（2026-07-11）**：`toSaveRecord` 先解析全部非 `field` 源、再解析 `field` 源
> （如 `acceptCodeLines`→引用 `acceptResult`），field 引用与映射键顺序无关。曾因 settings.json 里
> field 源丢 `path`（早期 UI 保存吞掉）导致计数恒 0 上送——`settings.js` load() 已做自愈迁移
> （缺 path 补 `acceptResult`、字段缺失按默认补）。
>
> ⚠️ **整数字段必须带数值 transform**：`codeLines`/`codeSize`/`promptSize` 若映射成裸 `resText`/`reqText`（漏 transform），会把整段 SSE 原文（字符串）发给接口的 Integer 字段 → 接口 `Cannot deserialize Integer from String "data:..."` → 整包被拒。手动加字段易踩，**用「重置为预设」**（预设已带正确 transform）。详见 [字段映射分析.md](字段映射分析.md)。

### `src/settings.js` — 设置管理

- `appTitle`：默认 `'资金同业代码解析工具'`，可通过面板修改
- `appVersion`：版本号，显示在页头中间，可通过面板「上送设置」修改（仅管理员）
- `umAccount`：自动取系统用户名（**小写**）；手动改后设 `umAccountManual=true` 持久化（防拷贝到别的机器）
- `reporterTargets.saveRecord.match`：默认 `'*/completions*'`（同时匹配两个接口）
- **映射自动迁移（2026-07）**：`load()` 时若保存的 `saveRecord.mapping.result` 仍是旧默认
  `{source:'resText',transform:'sseContent'}`，自动升级为 `sseFullReply`。原因：改 DEFAULT_MAPPING
  对已分发机器无效——它们的 settings.json 存着旧映射会覆盖默认值（教训：**改默认映射必须配迁移**）。
  迁移只在内存、幂等，下次「保存设置」落盘；用户自定义过的 result 取法不受影响。
- `debugLogEnabled`：调试落盘(JSONL)开关（默认关，仅管理员）。见下 `src/debug-log.js`。

> ⚠️ **多环境是必须考虑的事项（改配置/映射时的检查清单）**。本工具的配置有三层来源，改动要同步：
> 1. `src/config.js`：`process.env.*` 默认值（env 变量兜底，如 `UPDATE_CHECK_URL`）；
> 2. `src/settings.js` `defaults`：面板可改项的出厂默认（如 `debugLogEnabled`、`updateCheckUrl`）；
> 3. **`env/settings-fat.json` / `env/settings-fat001.json`：每个环境预置的完整 settings.json**，
>    `dist-release.cmd` 发布时注入成 dist 的 `settings.json`。
>
> **关键坑**：env 文件里的 `reporterTargets.*.mapping` 会**整体覆盖** `DEFAULT_MAPPING`（`toSaveRecord`
> 用保存的映射优先），且 `load()` 是 `{...defaults, ...存盘JSON}`——**存盘 JSON 缺的键才回落 defaults**。
> 所以：
> - 改了 `DEFAULT_MAPPING`（新增字段 / 换 transform）→ **必须同步改两个 env 文件的 mapping**，否则
>   fat/fat001 上该改动完全不生效（运行时迁移只兜 `result`，不补新增字段如 `acceptCodeLines`）。
> - 新增 `defaults` 里的配置项（如 `updateCheckUrl`）→ 若该项**每环境不同或需默认开启**，要在两个 env
>   文件里显式写值（不写则回落 defaults，可能是空/关）。
> - 改完 env 文件用 `node -e 'require("./env/settings-fat.json")'` 验证是合法 JSON。
> - env 文件含 token、被 `.gitignore` 忽略、不进 git/dist（仅 `dist-release.cmd` 注入）。填写说明见
>   [env/README.md](../env/README.md)。

### `src/update-check.js` — 新版本检查

- `GET /api/update/check`（`?force=1` 跳过缓存）：Node 侧 fetch `settings.updateCheckUrl`
  指向的内网接口/静态文件（`{version, downloadUrl, notes}`），与本地版本比较。
  走后端的原因：无 CORS/混合内容问题，检查地址不暴露给页面。
- **版本比较对格式宽容**（适配发版习惯"页头版本号末位累加"）：两侧都先 `extractVersion`
  提取点分数字段（`fat001- v1.0.2` → `1.0.2`），逐段**数值**比较（`1.0.9 < 1.0.10` 正确）。
  本地版本优先取 `settings.appVersion` 的数字段，提取不出才退回 `package.json` version。
- **外呼窗口/自查间隔跟随 `checkIntervalSeconds`**（`intervalMs`，下限 10 秒、默认 2h）：`check()` 的缓存 TTL
  与后端自查间隔都用它。意义——**后端任何变更（延长/撤销 `disableAt`、改 `minVersion`/`notes` 等）在一个间隔内
  即被拿到并切换横幅状态**（红↔橙↔蓝，`renderUpdateBanner` 每次 check 都按最新 `u` 重渲）；调小间隔=反映越快，
  手动「检查更新」`force=1` 永远即时。拉取失败静默返回 `{error}` 并 fail-open 保持上次管控态。
- 前端（index.html）：加载时查一次 + **可配间隔轮询**（`UPD_INTERVAL_MS` 默认 2h，被返回的 `checkIntervalSeconds` 覆盖、下限 10 秒；自重排 `setTimeout` `scheduleNextUpdCheck`）；有新版**常驻**弹页头蓝色横幅（不提供「不再提醒」，刷新仍在、升级自动消失；DOM API 组装防注入），
  有新版即常驻显示（不再提供「不再提醒」）；页头「检查更新」按钮 = force 强制刷新。
- `updateCheckUrl` 仅管理员可改（防止指向恶意下载源）；环境变量 `UPDATE_CHECK_URL` 可作打包默认值。
- **强制停用（minVersion + 可选 disableAt）**：返回数据带 `minVersion`，本地版本低于它 = "受管控"。
  - `disableAt`（绝对时刻，首选"年月日时分秒" `2026-07-15 00:00:00` 按本机时区、经 `parseDisableAt`
    显式构造本地时间归一到 ms；也兼容时间戳）：受管控时到此刻才真正停用，之前橙色倒计时横幅；
    不下发则立即停用（旧行为）。
  - `isDisabled()` = 受管控 && (无 disableAt || `Date.now() >= disableAtMs`)，**每次实时比 now**——
    倒计时到点无需重新 check、也不受重启影响即自动生效（存绝对时刻而非剩余秒是关键）。
  - 停用后 `proxy.js onRequest` 对新请求一律 503（不产生抓包记录），**上送队列不受影响**（已入队照发）。
  - 前端三态横幅：已停用(红) > 倒计时(橙,本地 setInterval 每秒 tick、到点切红) > 有新版(蓝)，均常驻不可关。
  - 检查失败 fail-open（保持上次状态，从未成功=不管控）；模块内启动 **2 秒**后首查、之后按 `intervalMs`
    自重排后台自查（`scheduleSelfCheck`，`unref`，不依赖面板开着）——故**面板没开时代理也能在一个间隔内
    感知后端变更**（如延长 `disableAt` → 自动解除 503、切回倒计时）。撤销 `minVersion`/升级后自动恢复。
- **发版只需改「上送设置」的版本号**（页头显示串，比较基准即其数字段）；`package.json` 仅兜底。

### `src/debug-log.js` — 调试落盘 JSONL（仅管理员）

- 面板「上送设置」开启 `debugLogEnabled` 后，`reporter.push()` 里每条**确定要上送**的记录会同步
  追加到项目根 `upload-debug-log.jsonl`（在 push 的内容过滤处，已算好 body，顺带写，不重复计算；
  因 `enqueuedIds` 去重每条只写一次；写失败仅告警，不影响上送）。
- 格式：**JSONL**，每行一条 JSON（换行/逗号等一律转义进字符串，永不错位，grep/jq/pandas 皆可）。
  字段：`time`(ISO) `id`(=抓包页记录 id，可搜索对应报文) `requestId` `prompt` `result` `acceptResult`
  `codeLines` `codeSize` `acceptCodeLines` `acceptCodeSize`(与上送同值，便于核对代码计数)
  `requestRaw`(请求完整报文体) `responseRaw`(响应完整报文体/SSE 原文，供取证复现代码提取)。
  键顺序由调用方 row 决定（JSON.stringify 保留插入顺序）。
- 下载：`GET /api/debug/log`（`requireAdmin`，含代码明文，绝不下发给非管理员）；面板有「下载 JSONL」按钮。
- ⚠️ 会把 prompt/产出代码/请求报文**明文写盘**（敏感），定位同 `DEBUG_UPLOAD`——本机排查工具，默认关，
  用完请关。文件不进分发包（pack.cmd 白名单式拷贝，根目录文件不被打包）。
- 早期曾用 CSV，因 prompt/代码/报文是大段多行文本，Excel 里一格塞几百行不便查看，改用 JSONL。

### `src/admin.js` + `src/admin-pubkey.js` — 管理员权限

- 「上送设置」「字段映射」两个 tab 仅管理员可见，基于 **Ed25519 离线签名令牌**（不依赖本地白名单）。
- 公钥内置 `src/admin-pubkey.js`；私钥 `secrets/admin-private-key.pem` 仅分发者本机、绝不分发。
- 令牌绑定系统用户名（大小写不敏感）+ 带过期时间；解锁后仅内存会话（重启失效）。
- 详见 [管理员权限.md](管理员权限.md)。

| 纯工具调用轮（旁白为空）整轮不上送 | 同上：`result=''` 被内容过滤（result/acceptResult 任一为空即丢）丢弃 | 同上——工具代码进 result 后非空，恢复上送 |
| 同轮"写文件+旁白贴代码"丢旁白代码 | `extractGeneratedCode` 有工具代码即 return（短路） | `reporter-mapping.js` → 改为工具代码 ∪ 旁白围栏并集（按文本去重） |
| 改了 DEFAULT_MAPPING 但已分发机器不生效 | settings.json 里 UI 保存过的旧映射覆盖代码默认值 | `settings.js` → `load()` 自动迁移 result `sseContent→sseFullReply`（幂等、只动旧默认形态） |

| 定位会清空筛选 | `locateTraffic` 无条件清筛选 | `index.html` → 仅当目标被筛掉才临时清 |
| 最近送达定位不高亮 | 历史行无 id | `index.html` → 历史行加 `hrow-{qid}`，`locateQueue` 按状态选行 |

## 五、前端关键功能（`web/public/index.html`）

- **抓包列表**：实时更新，右键菜单（查看详情 / 补送），倒序排列，"抓包#"列含 ↗ 定位链接（**按上送状态着色**：待送蓝/发送中黄/失败红/已送达绿）
- **报文详情**：请求体/响应体支持格式化、语法高亮（Web Worker）、复制，最大化显示
- **上送队列**：待处理 / 最近送达，点击行预览报文，显示上送接口返回状态及响应体；**每行有单条「重试」按钮**
- **两边互相定位**：↗ 定位时**保留当前筛选**（仅当目标被筛掉才临时清筛选）；已送达定位到「最近送达」行（`hrow-`）高亮
- **补送功能**：对已完成但未入队的报文，右键「补送」手动入队
- **上送设置**（仅管理员）：URL、格式（raw/behavior）、Token、字段映射、URL 过滤白名单、软件名称、版本号
- **页头**：中间显示版本号；右上角管理员解锁 🔒 / 签发 🔑（仅本机有私钥时）/ 倒计时
- **敏感信息掩码**：报文详情里 `apiToken` 始终掩码；**上送请求地址仅管理员可见**
- **局域网访问开关**（仅管理员，默认关，重启生效）：关=绑 `127.0.0.1` 只回环；开=绑 `0.0.0.0` 放开局域网（面板+代理）。实际暴露时页头红色横幅 + 控制台红字警告。实现 `settings.bindHost()`，详见 [管理员权限.md](管理员权限.md)
- `setPreText(preEl, text)`：`\r\n`→`\n` 规范化 + `unescapeForDisplay` 显示转义序列
- `uploadResponseHtml(resp)`：渲染上送接口返回（绿色=成功，红色=失败）

## 八、打包部署

```bat
pack.cmd        :: 生成 dist\proxy-inspector\，包含所有运行时文件
```

打包内容：`src/`、`web/`、`node_modules/`、`mapping-presets/`、`patches/`、`scripts/`、`package.json`、`start.cmd`

**排除**：`certs/`、`secrets/`（管理员私钥）、`scripts/gen-admin-token.js`、`scripts/gen-keypair.js`（签发/造钥脚本）、`settings.json`、`reporter-queue.json`

收件方解压后直接运行 `start.cmd`（需本机有 Node.js）。

> 想让某台**你信任**的 dist 副本也能签发管理员令牌（显示 🔑 按钮）：把 `secrets/admin-private-key.pem`（及可选的 `scripts/gen-*.js`）拷进该副本项目根。普通使用者机器**绝不要**放 `secrets/`。详见 [管理员权限.md](管理员权限.md)。

## ★ 字段映射/上送 回归测试：behavior-validator

**`D:\code\behavior-validator`**：极简 Spring Boot 校验器（从内网 `user-behavior-track` 的 DTO/校验提炼），
不依赖内网/DB，专门验证 proxy-inspector 的字段映射与取值类型是否被埋点接口接受。

**约定**：**以后凡改动上送/字段映射相关逻辑，先把「上送设置」的上送地址指向 `http://127.0.0.1:18080/api`
回归验证**，在「上送队列」看每条报文能否被接住（红色失败会显示 Jackson 根因，如 `Cannot deserialize
Integer from String ... codeLines`），确认无误再切回真实埋点地址。详见 [字段映射分析.md](字段映射分析.md) 与 `behavior-validator/README.md`。
