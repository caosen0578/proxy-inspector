# 报文 ↔ 埋点接口字段分析

> 数据源：`codebuddy HTTP 请求响应报文部分.txt`（CodeBuddy IDE 4.2.5 → glm-5-general，SSE 流式）
> 目标接口：用户行为埋点 v1.0 的 `saveRecord` 与 `updateRecordForAccept`

## 一、报文里能拿到的原料

### 1.1 请求头（可直接取）

| 报文头（小写后） | 值（样例） | 可填埋点字段 |
|---|---|---|
| `x-ide-type` | CodeBuddyIDE | **scope** |
| `x-ide-version` / `x-product-version` | 4.2.5 | **pluginVersion** |
| `x-user-id` | 6f3e36d4-…-30876038d332 | **createdBy** |
| `x-conversation-id` | 0b0bf59ba7ca45fa9358aa60fa7f5c0 | **sessionId** |
| `x-request-trace-id` | 65bebef4-…-a50ea213d6d1 | **requestId**（候选①）|
| `x-conversation-request-id` | e4606e099fad42f096ccalff11555ca7 | **requestId**（候选②）|
| `x-enterprise-id` / `x-tenant-id` | copilot | 备用（企业/租户）|
| `x-stream-intent` | craft | **type** 的推断依据（craft→?）|
| `monitor_prepareStartTime` | 1780534771329 | 客户端准备耗时计算 |
| `monitor_httpSendTime` | 1780534772967 | 客户端发送耗时计算 |

> ⚠️ 注意：本报文**只有 `X-IDE-Type`，无 `X-IDE-Name`**；有 `X-Conversation-Request-ID`，但**无 `X-Conversation-Message-ID`**。映射已据此把 scope 改为取 `x-ide-type`。

### 1.2 请求体（JSON）

```json
{ "model": "custom:glm-5-general", "max_tokens": 8192, "temperature": 0.6,
  "messages": [ {"role":"system","content":"…"}, {"role":"user","content":"…"} ] }
```

| 来源 | 可填字段 |
|---|---|
| `model` = `custom:glm-5-general` | modelName（候选：请求侧）|
| `messages[]` 拼接 | **prompt** |

### 1.3 响应（SSE 流式，关键发现）

响应头 `X-Request-Id: abece701-…`。响应体为 `data: {chunk}` 流，每个 chunk 的 `choices[0].delta` 有两类增量：

- `delta.reasoning` → **思考链**（应忽略，不算最终回答）
- `delta.content` → **正文**（拼接后即最终回答）

且**每个 chunk 都带 `usage`**（累计值），末尾 `data: [DONE]`。

| 提取方式 | 可填字段 | 实测结果 |
|---|---|---|
| 拼接所有 `delta.content`（忽略 reasoning）| **result** / acceptResult | 你好！我可以帮你处理代码相关的任务。有什么我可以协助你的吗？\n*本内容由AI生成* |
| 任一 chunk 的 `model` | **modelName** | glm5-0 |
| 末个含 usage 的 chunk `usage.prompt_tokens` | **promptTokens** | 8662 |
| `usage.completion_tokens` | **completionTokens** | 74 |
| `usage.total_tokens` | **totalTokens** | 8736 |
| 末个非空 `finish_reason` | **finishReason** | stop |

> ✅ **重要结论**：之前悬而未决的「token 三项是否拿得到」——**拿得到**，该模型每个 chunk 都返回 usage。必填的 promptTokens/completionTokens/totalTokens 可正常填充。

## 二、saveRecord 字段可填性总览

### ✅ 可从报文/代理直接填（15 项）

| 字段 | 来源 |
|---|---|
| createdBy | 头 `x-user-id` |
| sessionId | 头 `x-conversation-id` |
| requestId | 头 `x-request-trace-id`（待定，见下）|
| scope | 头 `x-ide-type` |
| pluginVersion | 头 `x-ide-version` |
| prompt | 请求 `messages` 拼接 |
| result / acceptResult | SSE `delta.content` 拼接 |
| modelName | SSE `model` |
| promptTokens / completionTokens / totalTokens | SSE `usage.*` |
| finishReason | SSE `finish_reason` |
| apiUrl | 请求 URL |
| cost | 代理测得响应耗时 `duration` |
| apiStatusCode / clientResponseCode | HTTP 状态码（语义待确认）|

### ⚙️ 可计算/固定（按需开启）

| 字段 | 方案 |
|---|---|
| type | 固定 `CODE_CHAT`，或按 `x-stream-intent`(craft) 映射 |
| triggerType | 固定 `auto` |
| isStatistics | 固定 `0`/`1`（默认值待定）|
| promptSize | 对 prompt 算字节数（可加 transform）|
| promptMd5 | 对 prompt 算 MD5（可加 transform）|
| waitCost / requestWaitCost | 由 `monitor_httpSendTime - monitor_prepareStartTime` 等计算 |

### ❌ 报文取不到（IDE 端/用户操作，建议留 null/空串）

`language`、`batchNo`、`commandType`、`codeLines`、`codeSize`、`repository`、`filePath`、`templateUuid`、`modelScope`、`instructionPath`、`knowledgeUuid`、`usage`、`isUseCache`、`resultId`、`acceptCodeLines`、`acceptCodeSize`

## 三、updateRecordForAccept 分析

| 字段 | 能否从本报文取 |
|---|---|
| requestId | ⚠️ 仅能靠与 saveRecord 同一标识关联 |
| actionType（codeCopy/codeAccept/codeEdit）| ❌ |
| acceptResult | ❌ |
| acceptCodeLines / acceptCodeSize | ❌ |

**结论**：updateRecordForAccept 的全部业务字段来自**用户在 IDE 内对生成结果的后续操作**（复制/采纳/编辑），这些事件**不出现在 AI 请求/响应 HTTP 报文中**。仅靠抓包**无法实现**该接口，需要 IDE 插件侧的操作事件作为数据源。

## 四、仍需你确认

1. **requestId** 用 `x-request-trace-id` 还是 `x-conversation-request-id`？（要能和后续 updateRecord 对上）
2. **modelName** 用响应的 `glm5-0` 还是请求的 `custom:glm-5-general`？
3. **type**：固定 `CODE_CHAT`，还是按 `x-stream-intent` 映射枚举？
4. **isStatistics** 默认 0 还是 1？
5. **apiStatusCode/clientResponseCode**：要 AI 业务码还是 HTTP 码？（SSE 流里没有业务码）
6. **acceptResult**（saveRecord 必填）：抓包阶段用户尚未"接受"，是否就等于 result？
7. **reasoning 思考链**要不要单独上送（目前只取 content 正文，丢弃 reasoning）？
